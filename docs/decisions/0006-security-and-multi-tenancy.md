# ADR 0006: Security and multi-tenancy

> Status: Accepted (design) · Implementation: Week 13 · Date: 2026-05

## Context

Through week 5 the system is single-user by construction: anyone who
can reach `/ask` can ask anything, and every question runs against the
same Postgres database with the same connection-time privileges. That
is fine for a portfolio demo on `localhost` but is the obvious next
hurdle for any "enterprise-credible" framing of the project.

This ADR sets the strategy **before** the week-13 implementation so
the reasoning is preserved separately from the eventual mechanical
code. The shape mirrors the earlier ADRs ([0004](0004-self-healing-policy.md),
[0005](0005-conversation-persistence.md)).

## Threat model

### In scope

* Untrusted callers hitting `/ask`. No external network sits in front
  of the API — anyone with the URL can submit a question, today.
* Multi-tenant data isolation. A future `acme-corp` tenant must not
  see `globex-llc`'s rows even with a deliberately crafted prompt.
* LLM-generated SQL bypassing visibility rules. The retriever +
  generator pair is purpose-built to produce arbitrary SQL. We cannot
  rely on it to honour any privacy invariant; the boundary belongs
  somewhere it cannot reach.
* Cost runaway. A single user can today loop the agent, send
  thousand-token prompts, and burn DeepSeek + SiliconFlow credit with
  no ceiling.
* Abusive prompt patterns. Same root cause as cost runaway: there is
  no per-user / per-IP throttle.

### Out of scope (for this ADR)

* High availability, DDoS resilience, hot-failover. Infra-layer
  concerns, addressed separately when the project ships to Fly.io.
* GDPR procedures (right-to-erasure, data export, residency
  attestations). Deferred to ADR 0008 — implementable on top of the
  audit log designed here, but a process problem more than a code
  problem.
* PII detection and masking inside SQL results. Deferred to ADR 0007.
* Self-hosted IdP. We use JWT for the demo; a Keycloak / OIDC swap
  is mentioned in §"Future work" below but is a week-14 problem at
  earliest.

## Decisions

### AuthN — JWT via FastAPI's `OAuth2PasswordBearer`

A standard short-lived JWT (HS256, 60-minute expiry) issued from a
`POST /auth/login` endpoint after a hashed-password check. The
`/ask` handler gains a `Depends(get_current_user)` that decodes the
token, returns the `User` row, and 401s on anything malformed.

Alternatives considered:

* **Static API keys.** Simpler to issue but too coarse: every key is
  effectively a long-lived bearer credential with no expiry, no
  per-call scope, and no easy way to rotate. Easy to add LATER as a
  parallel auth mechanism for programmatic callers if needed.
* **Full OAuth2 with Google as IdP.** Adds a third-party dependency
  the portfolio doesn't need to demonstrate. A future swap to OIDC
  (any provider) is a one-class change in `auth/jwt.py` because we
  only ever read the JWT claims, not who issued them.

### AuthZ — three-role RBAC

```
admin   → can do anything; sees all tenants' data.
analyst → can ask data questions within their own tenant.
viewer  → can ask data questions but only see aggregated / safe
          views; destructive prompts (drop / delete language) are
          short-circuited before generation.
```

Roles live on `users.role`, are baked into the JWT claims, and are
enforced via a `Depends(require_role(Role.analyst))` helper. Three
roles is the minimum cardinality that lets us tell a useful story
about least-privilege and a maximum that does not invent governance
ceremony for a demo.

Alternative considered: **Attribute-Based Access Control (ABAC)**
with per-resource policies. The right answer for a real SaaS;
overkill for this project's scope. Worth a mention in interviews as
"the next layer of refinement if we needed it".

### RLS — Postgres `CREATE POLICY` + per-request DB role

This is the portfolio-defining decision.

```sql
ALTER TABLE customers ENABLE ROW LEVEL SECURITY;

CREATE POLICY tenant_isolation_customers ON customers
    FOR ALL
    USING (tenant_id = current_setting('app.current_tenant')::text);
```

At connection-acquire time we `SET ROLE` to a low-privilege Postgres
role (`copilot_runtime`) and `SET LOCAL app.current_tenant =
'<tenant_id_from_jwt>'` based on the JWT claims. From there forward
**every SELECT** — whether handcrafted by us, generated by the LLM,
self-healed after a retry, or even injected via a clever prompt —
is silently filtered by the policy. The LLM doesn't know about
`tenant_id`. It doesn't need to. The database does.

**This is the entire portfolio answer:**

> *"The trust boundary belongs in Postgres, not in the prompt."*

Alternatives explicitly considered and rejected:

1. **App-level WHERE-clause injection on the LLM-produced SQL via
   `sqlglot` AST manipulation.** Plausible at first glance: the
   safety layer already parses every SQL into an AST, so adding an
   `AND tenant_id = ?` to every `FROM` clause is "just another AST
   walk". Rejected because:
   * It does not compose with self-healing. If the rewrite produces
     malformed SQL that the safety layer reluctantly accepts but the
     DB rejects, the retry prompt now contains both the original
     question AND our tenant predicate — leaking the existence of the
     invariant to the model.
   * It is fragile under unusual SQL shapes: `WITH` CTEs, subqueries
     in `WHERE`, `UNION` branches with different table sets. Each
     edge case is a potential bypass.
   * It cannot defend against a prompt like "list `customers JOIN
     internal_customer_table USING (id)`" if the user knows or
     guesses an admin-only table name. RLS at the DB level catches
     this by definition because the role lacks `SELECT` on the table.
   The "trust boundary in the database" formulation eliminates the
   entire class of bypasses by construction.

2. **Schema-per-tenant.** Cleanest isolation but operationally
   heavy: every schema change has to be replayed N times, the
   retriever's `schema_embeddings` table grows N×, and cross-tenant
   admin queries become a tedious `UNION ALL` exercise. RLS gives
   95% of the isolation guarantee at 10% of the operational cost.

3. **Two physical databases (one per tenant).** Mentioned for
   completeness; same trade-offs as schema-per-tenant, even heavier.

### Multi-tenancy — `tenant_id` column on every business table

Northwind data ships under a single seeded tenant (`demo`). New
tenants are inserted by an `admin`-role API call that copies the
seed (or accepts an empty seed). Each business table (`customers`,
`orders`, `order_details`, `products`, …) gets:

```sql
ALTER TABLE <t> ADD COLUMN tenant_id TEXT NOT NULL DEFAULT 'demo';
CREATE INDEX <t>_tenant_idx ON <t>(tenant_id);
```

The `tenant_id TEXT` rather than `UUID` is intentional: human-
readable IDs make admin diagnostics in `psql` legible without joining
to a `tenants` table for every investigation. Performance overhead
of the extra column is negligible at this scale; the index is what
keeps RLS-filtered queries fast.

### Audit log — append-only `audit_log` table

```sql
CREATE TABLE audit_log (
    id           BIGSERIAL PRIMARY KEY,
    ts           TIMESTAMPTZ NOT NULL DEFAULT now(),
    user_id      INT REFERENCES users(id),
    tenant_id    TEXT,
    question     TEXT,
    sql          TEXT,
    row_count    INT,
    latency_ms   INT,
    error_class  TEXT,  -- NULL on success
    attempts     INT
);
```

A FastAPI middleware writes one row per `/ask`, regardless of
success or failure. Append-only enforced at the application layer
(no `UPDATE` / `DELETE` paths in code) and via a `REVOKE UPDATE,
DELETE ON audit_log FROM copilot_runtime` grant.

Two things this gets us for free:

* **Forensics.** Any "did the agent really tell user X about
  customer Y" question becomes a single SQL query.
* **Eval harness fuel.** Week 6's evaluation set can be seeded with
  real user questions from the audit log (consent permitting).

### Rate limit — `slowapi` middleware

Per-user 30 req / min, per-IP 100 req / min. Anonymous clients
(before login) limited to 5 req / min on `/auth/login` and `/health`
to make brute-force password attempts expensive.

### Cost cap — `users.monthly_token_budget`

Decremented by the actual prompt + completion tokens returned by the
DeepSeek + SiliconFlow responses (we already log these via LangSmith
trace metadata). Over-budget requests return `429 cost_cap_exceeded`
with a `Retry-After` header set to the start of the next month.

Admin can adjust budgets per user; default for `analyst` is 100k
tokens / month (enough for a few hundred normal questions at
DeepSeek pricing — ≈¥1 / month per user at portfolio volumes).

## Consequences

### Good

* The portfolio narrative becomes "I can take this from demo to a
  real multi-tenant SaaS". RLS via Postgres policies is the single
  most credibility-buying choice in the whole project because it
  shows the author understands where trust boundaries belong in an
  LLM-driven data tool.
* `/ask` becomes safe to expose on the public internet.
* The audit log doubles as the Week 6 evaluation harness's data
  source.
* Adding new tenants becomes a one-row `INSERT` instead of a
  schema-per-tenant ceremony.

### Bad / accepted trade-offs

* Quickstart in README grows a `register + login` step; demos need
  a pre-seeded `admin@demo` user. We document this clearly and
  ship the seed in `data/seed/03-users.sql`.
* Northwind schema gains a `tenant_id` column on each business
  table — a slight deviation from the canonical sample data, but
  unavoidable for the demo to mean anything.
* JWT secret rotation is not built in; the secret lives in
  `.env`. Acceptable for a portfolio demo; non-acceptable for a
  real deployment, which would inject it from a secret manager
  (mentioned in §"Future work").
* Refresh tokens are deferred. 60-minute JWT expiry → users
  re-login. Fine for a demo, painful for a real product.

### Notable subtleties

* **Per-request `SET ROLE` matters.** If we used a connection-level
  privileged role and only relied on `SET app.current_tenant`,
  any leak of the privileged connection (a misplaced `DO` block,
  a forgotten `RESET`) would break tenant isolation. By dropping
  to `copilot_runtime` on every borrow, the privileged role is
  exercised only for migrations and admin paths.
* **Self-healing loop sees the same restriction.** The retry
  prompt template never sees `tenant_id` because the LLM never
  sees `tenant_id`. The DB silently filters rows it cannot read.
  If the LLM's bad SQL produces zero rows, the retry prompt
  honestly says "the previous SQL returned zero rows" — which is
  exactly the right level of detail.
* **Cross-tenant admin queries** require a one-off
  `SET LOCAL ROLE copilot_admin` inside an admin-tagged endpoint.
  We do NOT extend `/ask` to support an "act as tenant X" flag —
  that road leads to delegation bugs.

## Future work

* **ADR 0007 — PII detection and masking.** Row-preview to the
  summariser is currently raw values. A future ADR will design a
  column-tag + redact pipeline (`emails`, `phone_numbers`,
  `addresses` masked by default; admin can opt-in to raw).
* **ADR 0008 — GDPR procedures.** Right-to-erasure (tombstones +
  cascading deletes), data-export, region attestation. Sits on top
  of the audit log + RLS designed here.
* **Ops doc — HA + backup + secret manager.** Out of the ADR
  series; lives in `docs/ops.md` (written alongside week-11 deploy
  work).
* **Self-hosted IdP swap (Keycloak / OIDC).** One-class change in
  `auth/jwt.py`. Worth doing if the project ever grows past
  "portfolio demo" stage.
