# ADR 0014: Multi-agent — supervisor + analyst pattern

> Status: Accepted · Date: 2026-05 (Week 12.5) · Supersedes: partially overrides ADR 0001 §"single agent"

## Context

Through Week 12 the project deliberately ran as a **single LangGraph
state machine** with 12 specialized nodes. ADR 0001 §3 ("we do not
build a 'multi-agent' system") committed to that choice. Two follow-
up audits — one in interviews, one in usage — surfaced the same
gap: a fully linear answer pipeline cannot proactively surface
follow-up questions, drill-downs, or anomaly callouts. The
`Insight` envelope from Week 8 hints at it (`bullets`,
`metric_highlights`), but every bullet is generated by the same
`summarize_result_node` from the same rows the user already saw —
no second perspective, no "the agent looked harder at this."

Multi-agent collaboration is the canonical answer in 2026's agent
literature (Anthropic's research agents, AWS's Bedrock orchestrator,
LangChain's `Send` API). The trade-off ADR 0001 cited — debugging
overhead, agent-to-agent message drift, token cost — is real, but
*scoped* multi-agent (one supervisor + one specialised worker) buys
most of the upside while keeping the loop bounded.

## Decision

We introduce a **supervisor + worker pattern** with exactly two
worker agents:

* **SQL Specialist** — the existing 12-node graph, wrapped as a
  callable sub-graph. Unchanged code; just lifted under a parent.
* **Analyst** — new, three-node agent that runs *after* a successful
  data answer. Inspects the rows, the answer, and the conversation
  history; produces three structured outputs (anomalies, follow-up
  suggestions, optional drill-down request) on its own prompt /
  retry / token budget.

The **Supervisor** is the orchestrator. It is **not an LLM-driven
agent**; it is a thin LangGraph state machine with deterministic
routing:

```text
                  question / resume
                          │
                          ▼
                   ┌─────────────────┐
                   │  Supervisor     │
                   │  (rules only)   │
                   └─┬───────────────┘
                     │
                     ▼
              ┌──────────────────┐
              │  SQL Specialist  │   ◀──┐
              │  (week-12 graph) │     │drill-down loop
              └──┬───────────────┘     │ (hop_count ≤ 2)
                 │                     │
                 ▼                     │
        route_after_sql ──END (chitchat / single KPI / error)
                 │                     │
                 ▼ (data answer)       │
              ┌──────────────────┐     │
              │   Analyst        │ ───┘
              │  • anomalies     │
              │  • followups     │
              │  • drill_down?   │
              └──┬───────────────┘
                 │
                 ▼
        route_after_analyst ──END
                 │
                 ▼ (drill_down requested)
        loop back to SQL Specialist with the sharper question

```

State lives in a top-level `SupervisorState` TypedDict that includes
the full `AgentState` dict from the SQL Specialist as a nested
field — the specialist keeps its own state shape so its existing
nodes / tests / checkpointer interactions stay untouched.

Inter-agent messages are **Pydantic envelopes**
(`AnalystRequest`, `AnalystResponse`), not LangChain `BaseMessage`.
Reasons in §"Why Pydantic envelopes" below.

## Why supervisor+worker, not swarm / hierarchical / planner-executor

| Pattern | Fit for text-to-SQL | Notes |
|---|---|---|
| **Supervisor + worker** (chosen) | ✅ | Linear pipeline with one decision point ("does this answer need a second look?") |
| Swarm (peer-to-peer) | ❌ | No natural symmetry between SQL and Analyst — they have different I/O shapes |
| Hierarchical (manager + subordinates) | ⚠️ | Overkill for two agents; the manager IS the supervisor |
| Planner + executor | ❌ | Plan is fixed by the LangGraph topology; nothing to plan dynamically |
| ReAct / agent-as-LLM | ❌ | Replaces the entire week-1..12 architecture for marginal benefit |

Supervisor + worker is the smallest multi-agent pattern that ships a
genuine "agent collaboration loop" without recasting earlier work.

## Why the supervisor is rule-based, not LLM-driven

A common multi-agent pitfall is making the supervisor an LLM that
calls workers via tool-use. We deliberately avoid that:

1. **Cost**: every turn would pay one extra LLM round-trip just for
   "should I call the analyst?" — for a yes/no the LLM can answer
   from heuristics (row count, chart kind, error state).
2. **Determinism**: rule-based routing is unit-testable and
   reproducible across eval runs; LLM routing isn't.
3. **Observability**: every routing decision shows up as a
   deterministic LangGraph edge in LangSmith. Failure modes are
   bounded.

The supervisor's rules today (see `agents/supervisor.py`):

* `intent == "chitchat"` → END
* `row_count <= 1` and no chart-worthy structure → END
* `error` set → END (analyst can't analyse a failed turn)
* otherwise → forward to Analyst

The supervisor will evolve toward "rule-by-default, LLM-by-exception"
when the routing space grows; for now the rules cover every observed
case in the eval set.

## Why Pydantic envelopes, not LangChain BaseMessage

Inter-agent traffic is **structured data**, not "free-text messages":

* Analyst needs `rows`, `answer`, `chart_kind`, `dialogue_context` —
  fields with concrete types.
* SQL Specialist (when invoked recursively for a drill-down) needs
  the new `question` and the `parent_turn_id` for trace linking.

A `BaseMessage` blob would force every consumer to re-parse, lose
type safety, and bloat tokens. Pydantic models are the same shape
operators see on `/admin/stats` — one schema language end-to-end.

## Drill-down: bounded recursion

The Analyst's most powerful affordance is asking the SQL Specialist
a sharper question ("Germany has 11 customers — show me the ones
with no orders in the last 90 days"). This is real agent-to-agent
collaboration, but it's also where multi-agent systems run away.

We bound it strictly:

* `SupervisorState.hop_count` is incremented on every Specialist
  invocation.
* `route_after_analyst` checks `hop_count >= 2` before allowing
  another drill-down.
* The Analyst's `drill_down_node` itself refuses to emit a
  `drill_down_request` once `hop_count >= 1` (a single drill-down
  per turn is the demo-worthy upper bound).

This is the same "retry budget" pattern Week 4's self-healing loop
uses, applied to inter-agent calls.

## Eval coverage

`ANALYST_ENABLED` is a new feature flag (`feature_flags` already has
the pattern). The eval harness gets a fourth A/B
(`analyst_enabled.py`) that compares:

* **Baseline**: ANALYST_ENABLED=False, supervisor short-circuits after SQL.
* **Treatment**: ANALYST_ENABLED=True, full supervisor flow.

Metrics that change:

* `avg_total_tokens` — Analyst adds ~250-500 tokens per data turn.
* `success_rate` — should stay flat (Analyst doesn't gate the answer).
* New: `avg_followups_per_turn`, `avg_drill_downs_per_turn` — surfaced
  by the runner from the new `analyst` output field.

ADR 0007's deterministic grader stays unchanged; the Analyst's
output is additive, not graded for "correctness" in v1.

## API & state changes

`AskResponse` gains:

```python
analyst: AnalystResponse | None  # populated when supervisor invoked Analyst
drill_downs: list[AskResponse]   # populated when Analyst triggered drill-down
```

Both fields are backward-compatible (existing callers keep their
old contract; the analyst-aware caller opts in).

`/admin/stats` gains an `agents` block (number of analyst calls,
number of drill-downs, avg hop_count) so operators see whether the
new pattern is actually firing in production.

## Failure modes

| Scenario | Behaviour |
|---|---|
| `ANALYST_ENABLED=False` | Supervisor short-circuits; output identical to pre-12.5 |
| Analyst LLM call fails | Logged as warning; `analyst=None`; the original SQL answer still ships |
| Drill-down LLM proposes invalid SQL | Existing self-healing kicks in on the recursive Specialist call; if it fails, the drill-down is logged + skipped (does not block the parent answer) |
| `hop_count` blows past 2 | `route_after_analyst` returns END; we never recurse further |
| Supervisor receives `resume` (HITL) | Threaded through to the Specialist unchanged; Analyst is skipped on the resume turn (the user already saw the analyst output before approving) |

## Consequences

### Good

* Multi-agent collaboration loop with **bounded recursion** —
  portfolio story arrives without giving up debuggability.
* The single state machine still drives everything; LangGraph
  traces show the supervisor → specialist → analyst path
  deterministically.
* Week 8's `Insight` is no longer "the only second look at the
  data" — Analyst can disagree, drill in, or stay silent.
* Demo story: ask a question, watch the agent volunteer "you might
  also want to know …" without prompting.

### Bad / accepted trade-offs

* +250-500 tokens per data turn when Analyst fires. Documented in
  the eval A/B; operators can disable via `ANALYST_ENABLED=False`.
* Supervisor adds a layer of indirection: a user-visible bug ("why
  did the agent run twice?") now has three potential origins
  (Specialist, Supervisor routing, Analyst drill-down) instead of
  one. Tests pin the routes; LangSmith traces show the path.
* Recursive Specialist calls share the same conversation_id /
  thread_id with the parent — the dialogue gets two pairs of turns
  per visible "turn" (parent + drill-down). The `parent_turn_id`
  field on `AnalystResponse.drill_down` lets UI tooling group them.

## Future work

* **LLM-driven supervisor** when routing rules outgrow the
  decision table. Use `langchain.tools` so it stays auditable.
* **Analyst can call non-SQL tools** (e.g. ask Specialist to
  visualise a different slice instead of running new SQL). One more
  tool on the Specialist side; protocol stays the same.
* **Replace `Insight` with `analyst.summary`** in a future major
  version — `Insight` becomes redundant when Analyst always runs.
  Kept side-by-side for backwards compatibility through Week 12.5.
* **Critic agent** that grades Analyst outputs against the rows.
  Recursion budget would need to grow to 3.
